\# AGENTS.md



이 문서는 Codex가 이 프로젝트에서 항상 먼저 참고해야 하는 작업 규칙이다.



목표는 비개발자도 이해할 수 있는 방식으로, \*\*최소 변경 · 검증 중심 · 과도한 추측 없는 개발\*\*을 진행하는 것이다.

\## 1. 사용자와 프로젝트 맥락



사용자는 비개발자 출신이다.



따라서 Codex는 설명할 때 다음 원칙을 따른다.



\* 전문용어는 가능한 쉽게 풀어서 설명한다.

\* 작업 결과는 짧고 명확하게 보고한다.

\* “무엇을 바꿨는지 / 왜 바꿨는지 / 어떻게 확인했는지”를 반드시 구분한다.

\* 사용자가 요청하지 않은 기술적 확장을 임의로 하지 않는다.

\* 사용자가 이해하기 어려운 구조 변경은 먼저 이유를 설명한다.



사용자는 바이브코딩으로 수익화 가능한 마이크로 SaaS를 만들고 있다.



현재 주요 사용 서비스는 다음과 같다.



\* Vercel: 프론트엔드 배포

\* Heroku: 백엔드 배포

\* MongoDB: 데이터베이스

\## 2. 제품 핵심 원칙



이 프로젝트는 잡리스크(JobRisk)다.



\### 2.0 제품 전략 절대 원칙



아래 4가지는 잡리스크 관련 큰 판단에서 항상 먼저 확인한다.



\* 잡리스크는 회사를 좋은 회사/나쁜 회사로 판정하는 서비스가 아니다.

\* 잡리스크는 채용공고를 요약하는 서비스가 아니라, 모호한 업무 문장을 검토 대상으로 바꾸는 서비스다.

\* 잡리스크의 핵심 가치는 공고 근거를 면접 질문과 답변 해석 기준, 최종 행동 가이드로 연결하는 데 있다.

\* 초기 핵심 고객은 스타트업·중소기업의 모호한 운영형/사무형 공고를 검토하는 취업·이직 준비자다.

\### 2.0.1 랜딩·카피·포지셔닝 상시 참조 문서

랜딩페이지, CTA, 카피, 상품 설명, 포지셔닝, 결과물 소개 문구를 수정할 때는 항상 먼저 아래 문서를 참고한다.

\* [docs/reference/landing-positioning-playbook.md](docs/reference/landing-positioning-playbook.md)

이 문서는 다음 3개 외부 참고 문서의 핵심을 잡리스크용으로 정리한 상시 가이드다.

\* `랜딩페이지-설계-원칙.txt`
\* `사이먼-사이넥-골든-서클-why-how-what (1).txt`
\* `마케팅-천재가-된-맥스.txt`

적용 원칙:

\* 기능 설명보다 사용자 문제와 구매 이유를 먼저 쓴다.
\* WHY -> HOW -> WHAT 순서를 유지한다.
\* 범용 AI 대비 차별점을 반드시 구조적으로 설명한다.
\* `공고 근거 -> 면접 질문 -> 답변 해석 -> 행동 결정` 흐름을 약화시키는 문구는 피한다.
\* 근거 없는 수치, 후기, 성과, 과장 표현은 금지한다.
\* 이 문서는 일반 마케팅 문서가 아니라 잡리스크 전략을 강화하기 위한 참고 문서다. 충돌 시 잡리스크 제품 원칙이 우선한다.



잡리스크는 채용공고를 읽고, 그 일이 사용자의 커리어에 자산이 될지 아니면 물경력으로 이어질 가능성이 높은지 미리 검토하도록 돕는 서비스다.



현재 프리토타입의 핵심은 넓은 의미의 모든 채용 리스크를 분석하는 것이 아니라, \*\*물경력 가능성 하나를 날카롭게 감지하는 것\*\*이다.



Codex는 모든 개발 판단에서 아래 원칙을 우선한다.



\### 2.1 잡리스크가 하는 것



\* 채용공고를 바탕으로 물경력 가능성을 검토한다.

\* 반복업무 비중, 의사결정 권한 및 자율성, 성과 측정 가능성, 난이도 상승 가능성, 시장 수요가 있는 범용 역량, 업무 범위 명확성/직무 전문성 집중도, 학습/피드백 구조를 중심으로 분석한다.

\* 공고에서 확인 가능한 근거를 바탕으로 위험 신호를 설명한다.

\* 사용자가 지원 전에 확인해야 할 질문을 제공한다.

\* 무료 결과에서는 짧고 날카로운 위험 신호를 보여준다.

\* 유료 결과에서는 면접에서 검증 가능한 질문과 답변 해석 기준을 제공한다.



\### 2.2 잡리스크가 하지 않는 것



\* 회사를 좋은 회사/나쁜 회사로 최종 판정하지 않는다.

\* 조직문화, 인간관계, 야근, 스트레스, 꼰대 여부 등을 공고만 보고 단정하지 않는다.

\* 외부 평판, 뉴스, 잡플래닛 후기 등을 최종 판정 근거로 삼지 않는다.

\* 공고에 없는 사실을 억지로 추론하지 않는다.

\* 사용자의 성향, MBTI, 워라벨 선호 등 개인화 기준을 현재 프리토타입의 핵심으로 확장하지 않는다.

\* 물경력 외 리스크로 제품 범위를 넓히지 않는다. 단, 사용자가 명시적으로 요청한 경우에는 먼저 계획을 제안한다.



\### 2.3 결과 표현 원칙



분석 결과는 단정형보다 검토형을 우선한다.



좋은 표현:



\* “물경력 위험이 높아 보입니다.”

\* “추가 확인이 필요합니다.”

\* “이 공고만으로는 단정하기 어렵지만, 반복 운영 업무 비중이 높아 보입니다.”

\* “면접에서 실제 기획/개선 권한이 있는지 확인해야 합니다.”

\* “이 표현은 업무 범위가 넓어 보이지만, 실제로는 잡무성 업무가 섞일 가능성이 있어 확인이 필요합니다.”

\* “반복 업무가 있더라도 개선 권한이나 성과 분석 구조가 있다면 경력 자산이 될 수 있습니다.”

\* “공고상 고용 형태가 불명확하므로 계약 조건을 별도로 확인해야 합니다.”



피해야 할 표현:



\* “이 회사는 나쁜 회사입니다.”

\* “지원하면 안 됩니다.”

\* “무조건 물경력입니다.”

\* “이 회사는 성장 가능성이 없습니다.”

\* “야근이 많을 것입니다.”

\* “운영 업무가 있으니 물경력입니다.”

\* “계약직이므로 무조건 커리어에 도움이 안 됩니다.”

\* “레거시 기술을 쓰므로 지원하면 안 됩니다.”

\## 3. Codex 기본 작업 원칙



\### 3.1 애매하면 바로 구현하지 않는다



작업이 모호하면 먼저 다음을 정리한다.



\* 내가 이해한 요청

\* 필요한 가정

\* 헷갈리는 지점

\* 가능한 선택지

\* 추천하는 방향



단, 사소하고 명확한 수정은 불필요하게 질문하지 말고 진행한다.



\### 3.2 요청 범위를 넘지 않는다



Codex는 사용자가 요청하지 않은 기능을 추가하지 않는다.



금지 사항:



\* 관련 없는 리팩토링

\* 주변 코드 스타일 임의 변경

\* 파일 구조 대규모 변경

\* 기능 범위 확장

\* 새 라이브러리 임의 추가

\* 인증, 결제, DB 스키마, 환경변수 임의 변경



발견한 문제 중 요청 범위 밖의 것은 수정하지 말고 마지막에 “참고 사항”으로만 적는다.



\### 3.3 최소 변경을 우선한다



문제 해결에 필요한 파일과 줄만 수정한다.



원칙:



\* 기존 코드 스타일을 따른다.

\* 단일 용도 코드에 과한 추상화나 설정화를 추가하지 않는다.

\* 한 파일 안에서 해결 가능한 문제를 여러 파일로 확장하지 않는다.

\* 새 라이브러리 추가는 정말 필요한 경우에만 한다.

\* 새 라이브러리가 필요하면 먼저 이유와 대안을 설명한다.



\### 3.4 코드 수정 전 계획을 먼저 제안한다



분석 로직, 결제, 인증, DB, 배포, 주요 UI 변경 작업에서는 바로 수정하지 않는다.



특히 아래 작업에서는 전략 적합성도 함께 점검한다.



\* 분석 로직 변경

\* 랜딩/카피 변경

\* 유료 결과 구조 변경

\* 제품 포지셔닝/상품 설명 변경



먼저 아래 형식으로 계획을 제안한다.



```md

\\## 이해한 문제

\\- ...



\\## 관련 파일

\\- ...



\\## 현재 로직 추정

\\- ...



\\## 최소 수정 계획

\\- ...



\\## 전략 적합성

\\- 이 변경의 직접 대상 고객:

\\- 범용 AI 대신 잡리스크를 써야 하는 이유와 연결되는 지점:

\\- 공고 근거 → 면접 질문 → 답변 해석 → 행동 결정 흐름 중 강화되는 부분:

\\- 제품 범위 이탈 여부:



\\## 위험한 부분

\\- ...



\\## 검증 방법

\\- ...

```



사용자가 명시적으로 “바로 구현해”라고 한 경우에도, 위험한 변경이면 먼저 짧게 계획을 보고한다.

\## 4. 잡리스크 분석 로직 작업 규칙



잡리스크의 분석 로직은 제품의 핵심이다.



따라서 분석 로직을 수정할 때는 단순히 코드가 동작하는지만 확인하지 말고, 분석 결과 품질이 좋아졌는지 확인해야 한다.



\### 4.0 잡리스크 7대 핵심 판정축



잡리스크의 물경력 분석은 기존 5개 축을 고도화한 7대 핵심 판정축을 고도화한 아래 7개 핵심 판정축을 따른다.



\#### 1. 반복업무 비중



단순 수행, 등록, 업로드, 검수, 정리, 응대, 요청 처리 중심인지 확인한다.



단, 반복 업무가 있다고 무조건 위험으로 판단하지 않는다. 반복 업무라도 개선, 자동화, 분석, 성과 관리, 프로세스 효율화로 이어지면 물경력 위험을 낮춘다.



위험 신호:



\* 단순 등록

\* 업로드

\* 검수

\* 정리

\* 반복 응대

\* 요청 처리

\* 정산 보조

\* 내부 시스템 입력



긍정 신호:



\* 반복 업무 개선

\* 자동화

\* 프로세스 효율화

\* 데이터 분석

\* 성과 기반 개선

\* 오류율 감소

\* 처리 시간 단축



\#### 2. 의사결정 권한 및 자율성



기존의 “책임 범위”보다 더 중요한 기준이다.



단순히 담당하는 일이 많은지가 아니라, 사용자가 스스로 판단하고 결정할 수 있는 영역이 있는지 확인한다.



위험 신호:



\* 지원

\* 보조

\* 어시스트

\* 요청사항 반영

\* 상급자 지시 수행

\* 정해진 프로세스에 따른 처리

\* 운영 지원

\* 내부 요청 대응



긍정 신호:



\* 전략 수립

\* 우선순위 결정

\* 개선안 도출

\* 실험 설계

\* 문제 정의

\* 의사결정 참여

\* 기획부터 실행까지 담당

\* 성과 기반 개선



주의할 점:



\* “총괄”, “담당”, “관리”라는 단어가 있어도 실제 의사결정 권한이 보이지 않으면 추가 확인이 필요하다.

\* 책임 범위가 넓어 보여도 실행만 하는 역할이면 물경력 위험이 높아질 수 있다.



\#### 3. 성과 측정 가능성



이 업무가 나중에 이력서나 면접에서 성과로 설명 가능한지 확인한다.



위험 신호:



\* 수행 업무만 나열되어 있음

\* KPI, 목표, 성과 기준이 없음

\* 처리량 외 성과 기준이 없음

\* “업무 전반”처럼 결과 기준이 모호함



긍정 신호:



\* KPI

\* 전환율

\* 매출

\* 리텐션

\* ROAS

\* CAC

\* 비용 절감

\* 품질 개선

\* 처리 시간 단축

\* 고객 만족도 개선

\* 오류율 감소



\#### 4. 난이도 상승 가능성



시간이 지나며 더 큰 문제, 더 넓은 범위, 더 높은 책임으로 확장될 수 있는지 확인한다.



위험 신호:



\* 처음 배운 뒤 같은 업무만 반복될 가능성이 큼

\* 업무 난이도 상승 구조가 보이지 않음

\* 성장 가능성은 언급하지만 실제 역할 확장 근거가 없음



긍정 신호:



\* 프로젝트 리딩

\* 더 큰 예산 또는 채널 담당

\* 문제 해결 범위 확대

\* 기획/분석/개선 역할로 확장

\* 주니어에서 오너십 있는 역할로 성장 가능



\#### 5. 시장 수요가 있는 범용 역량



기존의 “전이 가능한 역량”을 더 명확히 표현한 기준이다.



그 회사 내부에서만 통하는 경험인지, 다른 회사에서도 인정받을 수 있는 역량인지 확인한다.



위험 신호:



\* 특정 회사 내부 시스템만 다룸

\* 회사 고유 프로세스에만 익숙해지는 업무

\* 외부 이직 시장에서 설명하기 어려운 업무

\* 범용 툴, 지표, 방법론이 보이지 않음



긍정 신호:



\* 다른 회사에서도 쓰이는 직무 역량

\* 시장에서 수요가 있는 툴 또는 방법론

\* 기획, 분석, 개선, 운영 최적화, 리딩 경험

\* 포트폴리오나 성과 수치로 설명 가능한 경험



\#### 6. 업무 범위 명확성 / 직무 전문성 집중도



업무 범위가 명확한지, 너무 여러 직무가 섞여 전문성이 흐려지지 않는지 확인한다.



위험 신호:



\* 기타 업무

\* 다양한 업무 지원

\* 전반적인 운영 업무

\* 필요 시 유관 업무 수행

\* 유연한 업무 대응

\* 직무와 무관한 업무가 함께 나열됨

\* 콘텐츠, CS, 정산, 총무, 마케팅, MD 등 서로 다른 업무가 한 역할에 과도하게 섞임



긍정 신호:



\* 역할 범위가 명확함

\* 핵심 직무가 분명함

\* 협업 범위와 본인 책임 범위가 구분됨

\* 여러 업무가 있더라도 하나의 전문성으로 연결됨



주의할 점:



\* 업무가 많다고 무조건 좋은 것이 아니다.

\* 여러 일을 경험하는 것처럼 보여도, 핵심 전문성이 쌓이지 않으면 물경력 위험이 높다.



\#### 7. 학습/피드백 구조



업무 결과를 보고 배우고 개선하는 구조가 있는지 확인한다.



반복 업무라도 결과 분석, 회고, 실험, 개선으로 이어지면 커리어 자산이 될 수 있다. 반대로 반복 업무가 처리 완료로만 끝나면 물경력 위험이 높다.



위험 신호:



\* 처리 건수 중심

\* 빠른 응대 중심

\* 단순 등록/검수/정산 중심

\* 결과 분석 없음

\* 회고나 개선 프로세스 없음

\* 상위자가 시키는 대로 수행



긍정 신호:



\* 성과 리뷰

\* 데이터 기반 개선

\* A/B 테스트

\* VOC 분석

\* 문제 원인 분석

\* 재발 방지

\* 프로세스 개선

\* 실험과 학습



\### 4.0.1 보조 체크 항목



아래 항목은 물경력 핵심 판정축은 아니지만, 공고에서 확인되면 별도 경고 또는 추가 확인 질문으로 제공한다.



\#### 1. 고용 형태 / 조직 내 포지션



계약 구조나 조직 내 포지션이 불명확하면 커리어 자산화에 영향을 줄 수 있다.



체크할 표현:



\* 계약직

\* 파견

\* 외주

\* 프리랜서

\* 인턴

\* 전환형

\* 정규직 전환 가능

\* 프로젝트 계약

\* 소속 조직 불명확

\* 클라이언트사 상주

\* 채용 형태 불명확



주의할 점:



\* 고용 형태만으로 물경력이라고 단정하지 않는다.

\* 물경력 점수의 핵심 근거로 삼기보다는 “고용 구조 확인 필요”로 분리한다.

\* 면접 또는 계약 전 확인 질문으로 연결한다.



예시 질문:



\* 이 포지션의 정확한 고용 형태는 무엇인가요?

\* 정규직 전환 가능성이 있다면 기준과 과거 전환 사례가 있나요?

\* 실제 소속 조직과 업무 지시 라인은 어디인가요?



\#### 2. 기술 및 프로세스 낙후도



사용하는 도구, 기술, 업무 방식이 시장에서 너무 뒤처져 있으면 경력 가치가 낮아질 수 있다.



체크할 표현:



\* 수기 관리

\* 엑셀 중심 반복 관리

\* 오래된 내부 시스템

\* 자동화 없음

\* 분석 도구 없음

\* 레거시 시스템 유지보수만 수행

\* 신규 개발 없이 장애 대응 위주

\* 데이터 기반 의사결정 부재



주의할 점:



\* 이 기준은 직무군별로 다르게 적용한다.

\* 개발 직군에서는 기술 스택과 레거시 여부가 중요하다.

\* 마케팅/운영 직군에서는 데이터 분석 도구, 자동화, 성과 추적 구조가 중요하다.

\* 공고에 도구나 프로세스 정보가 없으면 단정하지 말고 추가 확인으로 처리한다.



예시 질문:



\* 이 업무에서 주로 사용하는 도구나 시스템은 무엇인가요?

\* 반복 업무를 줄이기 위한 자동화나 프로세스 개선이 이루어지고 있나요?

\* 성과 분석이나 업무 개선에 사용하는 데이터가 있나요?



\---



\### 4.1 분석 로직 변경 시 반드시 지킬 것



분석 로직 변경은 아래 중 하나의 작은 단위로만 진행한다.



\* 무료 미리보기의 한 줄 결론 개선

\* 가장 강한 근거 1개 선택 로직 개선

\* 짧은 이유 2\~3개 생성 방식 개선

\* 확인 질문 생성 방식 개선

\* 유료 상세 분석의 7대 핵심 판정축 개선

\* 고용 형태 / 기술 및 프로세스 낙후도 보조 체크 개선

\* 면접 질문 생성 품질 개선

\* 답변 해석 가이드 개선

\* 공고에 없는 내용을 단정하지 않도록 표현 개선

\* 직무군별 물경력 해석 기준 개선



한 번에 전체 분석 프로세스를 대규모로 바꾸지 않는다.



\### 4.2 무료 결과 원칙



무료 결과는 짧아야 한다.



무료 결과의 목적은 모든 정보를 제공하는 것이 아니라, 사용자가 “이 공고는 한 번 더 확인해야겠다”고 느끼게 하는 것이다.



무료 결과는 가능하면 아래 구조를 따른다.



\* 한 줄 결론

\* 가장 강한 근거 1개

\* 짧은 이유 2\~3개

\* 확인 질문 1개



무료 결과에서 유료 상세 분석의 핵심 가치를 과도하게 노출하지 않는다.



\### 4.3 유료 결과 원칙



유료 결과는 단순히 무료보다 긴 설명이 아니어야 한다.



유료 결과의 핵심은 사용자가 실제 의사결정을 할 수 있게 만드는 것이다.



유료 결과는 가능하면 아래 내용을 포함한다.



\* 최종 요약 1문장

\* 직무군 기준 상세 판정

\* 7대 핵심 판정축 결과

\* 보조 체크 항목 결과: 고용 형태 / 기술 및 프로세스 낙후도

\* 핵심 근거 3\~5개

\* 면접에서 꼭 물어볼 질문 5\~7개

\* 답변 해석 가이드

\* 최종 행동 가이드



유료 결과는 “더 많은 설명”보다 “더 구체적인 확인 기준”을 제공해야 한다.

\## 5. 골든셋 기반 검증 규칙



잡리스크 분석 로직을 수정할 때는 가능한 경우 골든셋으로 전후 비교한다.



골든셋은 잡리스크가 반드시 잘 판단해야 하는 대표 채용공고 테스트 묶음이다.



\### 5.1 골든셋의 목적



골든셋은 다음을 확인하기 위한 기준 문제집이다.



\* 물경력 위험도 판정이 적절한가

\* 근거가 공고 원문에 기반하는가

\* 공고에 없는 내용을 단정하지 않는가

\* 무료 결과가 짧고 날카로운가

\* 유료 결과가 면접 검증에 도움이 되는가

\* 기존에 잘 되던 결과가 새 변경으로 망가지지 않았는가



\### 5.2 골든셋 변경 금지



Codex는 사용자의 명시적 요청 없이 골든셋의 기대 위험도, 기대 근거, 정답 라벨을 임의로 바꾸지 않는다.



골든셋의 정답 라벨은 제품 오너인 사용자가 확정하는 기준이다.



Codex는 골든셋 결과가 이상해 보이면 수정하지 말고 “라벨 검토 필요”로 보고한다.



\### 5.3 분석 로직 변경 후 비교 항목



분석 로직을 수정한 뒤에는 가능한 경우 아래 기준으로 전후 비교한다.



```md

\\## 골든셋 전후 비교

\\- 좋아진 샘플:

\\- 나빠진 샘플:

\\- 그대로인 샘플:

\\- 기대 위험도와 불일치한 샘플:

\\- 공고에 없는 내용을 단정한 샘플:

\\- 추가 수정이 필요한 부분:

```



골든셋 실행 환경이 아직 없다면, Codex는 구현 가능한 최소 검증 방법을 제안한다.

\## 6. 백엔드 / 데이터 작업 규칙



백엔드, API, MongoDB, 인증, 결제, 배포 관련 작업에서는 특히 보수적으로 작업한다.



\### 6.1 보안



\* API 키, 토큰, DB 주소, 비밀값은 코드에 직접 쓰지 않는다.

\* 비밀값은 `.env` 또는 배포 환경변수를 사용한다.

\* 에러 메시지에 민감한 정보를 노출하지 않는다.



\### 6.2 데이터



\* 기존 데이터 구조를 임의로 바꾸지 않는다.

\* DB 스키마나 응답 구조를 바꾸면 영향받는 프론트엔드 코드도 확인한다.

\* 마이그레이션이 필요한 변경은 먼저 계획을 제안한다.

\* 기존 사용자 데이터가 손상될 수 있는 작업은 임의로 진행하지 않는다.



\### 6.3 배포



\* Vercel과 Heroku의 환경 차이를 고려한다.

\* 로컬에서는 되지만 배포 환경에서 실패할 수 있는 부분을 확인한다.

\* 환경변수, CORS, API URL, 빌드 명령어, Node 버전 차이를 주의한다.

\## 10. Codex가 자주 사용해야 할 작업 방식



\### 10.1 분석 로직 개선 요청을 받았을 때



바로 구현하지 말고 아래 순서를 따른다.



1\. 관련 파일 찾기

2\. 현재 로직 요약

3\. 문제 원인 추정

4\. 최소 수정 계획 제안

5\. 검증 방법 제안

6\. 사용자 승인 후 구현



\### 10.2 큰 요청을 받았을 때



예를 들어 사용자가 “분석 프로세스를 개선해줘”라고 요청하면, 전체를 한 번에 바꾸지 않는다.



먼저 작업을 작게 쪼갠다.



예:



\* 무료 미리보기 개선

\* 근거 선택 로직 개선

\* 직무군 분류 개선

\* 면접 질문 개선

\* 과잉 단정 표현 개선

\* 유료 결과 구조 개선



그다음 한 번에 하나씩 진행한다.



\### 10.3 사용자가 CSV나 여러 공고를 제공했을 때



100개 공고를 한 번에 전부 분석하고 바로 코드 수정하지 않는다.



먼저 대표 샘플을 나눈다.



\* 고위험 물경력 후보

\* 중간/추가확인 후보

\* 낮은 위험 후보



그다음 골든셋 후보로 사용할 샘플을 제안하고, 사용자가 확정한 뒤 검증에 사용한다.



\### 10.4 크레딧을 아끼는 방식



\* 불필요한 파일 탐색을 줄인다.

\* 관련 가능성이 높은 파일부터 확인한다.

\* 작업 범위를 작게 유지한다.

\* 긴 설명보다 변경 파일, 변경 내용, 검증 결과 중심으로 보고한다.

\* 한 세션에서 너무 많은 문제를 동시에 해결하려 하지 않는다.
