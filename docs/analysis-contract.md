# JobRisk Analysis Contract v1 Draft

## 1. 목적

이 문서는 JobRisk의 실제 분석 API, OpenAI 프롬프트, deterministic fallback, 테스트가 공통으로 따라야 하는 분석 계약 초안이다.

이 문서는 `AGENTS.md`의 제품 원칙을 실행 가능한 판정 규칙으로 번역한다.

## 2. 적용 범위

- 채용공고 텍스트 기반 분석
- 무료 미리보기와 유료 상세 분석
- OpenAI 구조화 출력
- deterministic fallback
- 테스트 검증 기준

## 3. 최상위 원칙

- 공고에 없는 사실은 단정하지 않는다.
- 강한 판단은 반드시 공고 원문 인용 또는 구조화된 근거에 연결한다.
- 근거가 약하거나 충돌하면 더 낮은 확신으로 내린다.
- 근거 없는 위험 판정은 금지한다.
- 근거 없는 긍정 판정도 금지한다.
- 근거가 부족하면 기본값은 `추가 확인`이다.
- 회사 평판, 소문, 조직문화 추정은 금지한다.
- 회사 정보는 참고 정보이며 직무 근거를 대체하지 못한다.

## 4. 판정 모델

각 축과 보조 체크는 아래 두 층으로 해석한다.

1. `evidenceStatus`
2. `signalDirection`

이 둘은 서로 다른 개념이다.

- `evidenceStatus`는 근거가 얼마나 확인되었는지를 뜻한다.
- `signalDirection`은 그 근거가 긍정인지 위험인지 혼합인지, 아니면 방향을 정하기 어려운지를 뜻한다.

즉, 같은 `확인됨`이어도 긍정 신호일 수 있고 위험 신호일 수 있다.

## 5. evidenceStatus 정의

`evidenceStatus`는 판단 방향이 아니라 근거 상태다.

허용값은 아래 3개뿐이다.

- `confirmed`
  - 공고 원문 또는 구조화된 근거로 뒷받침되는 판단
  - 직접 인용 또는 명시적 표현이 있어야 한다

- `needs_verification`
  - 일부 신호는 있으나 단정 근거가 닫히지 않은 경우
  - 긍정/위험 신호가 함께 보이거나 표현이 넓고 모호한 경우
  - 해석 가능성은 있지만 면접 또는 추가 정보 확인이 필요한 경우

- `insufficient`
  - 공고만으로는 판단에 필요한 정보가 사실상 없는 경우
  - 의미 있는 인용을 붙여도 방향 판단까지 이어지지 않는 경우
  - 이 상태는 안전 의미가 아니라 정보 부족 의미다

## 6. signalDirection 정의

`signalDirection`은 근거의 방향이다.

허용값은 아래 4개뿐이다.

- `positive`
  - 경력 자산이 될 가능성을 뒷받침하는 신호

- `risk`
  - 물경력 가능성을 높이는 신호

- `mixed`
  - 긍정 신호와 위험 신호가 함께 있어 한쪽으로 닫히지 않는 신호

- `neutral_or_insufficient`
  - 방향을 정할 근거가 약하거나, 아직 해석보다 확인이 먼저 필요한 상태

## 7. evidenceStatus 와 signalDirection 조합 원칙

허용되는 대표 조합은 아래와 같다.

- `confirmed + positive`
  - 예: KPI 개선, 실험 설계, 결과 책임, 개선 권한이 공고 문구로 직접 확인됨

- `confirmed + risk`
  - 예: 반복 등록, 요청 처리, 단순 운영, 권한 부재, 성과 기준 부재가 공고 문구로 직접 확인됨

- `confirmed + mixed`
  - 예: 운영 업무와 개선 업무가 함께 명시되어 있어 한 방향으로 단정하면 과장되는 경우

- `needs_verification + positive`
  - 예: 긍정적으로 읽히는 표현은 있으나 실제 권한 범위나 KPI ownership이 닫히지 않은 경우

- `needs_verification + risk`
  - 예: 위험하게 읽히는 표현은 있으나 실제 반복업무 비중이나 개선 책임 범위가 닫히지 않은 경우

- `needs_verification + mixed`
  - 예: 혼합 신호가 있고 추가 질문 없이는 방향 결론이 어려운 경우

- `insufficient + neutral_or_insufficient`
  - 예: 공고 정보가 너무 짧거나 핵심 역할 설명이 부족한 경우

원칙:

- `strong_positive`와 `risk`를 둘 다 단순히 `확인됨`으로만 압축하지 않는다.
- 내부 richer state가 있더라도 외부 contract에서는 `evidenceStatus`와 `signalDirection`을 분리해 표현한다.
- `confirmed`는 좋다/나쁘다가 아니라 "근거가 닫혔다"는 뜻이다.

## 8. 최종 위험도 정의

최종 위험도는 아래 3단계만 허용한다.

- `low`
  - 핵심 축에서 확인된 긍정 신호가 충분하고
  - 강한 위험 신호가 핵심 축에서 우세하지 않으며
  - 남아 있는 확인 필요가 최종 해석을 뒤집을 정도로 핵심적이지 않을 때

- `needs_verification`
  - 기본값
  - 근거 부족, 혼합 신호, 핵심 축의 모호성, 핵심 판단에 영향을 주는 확인 필요가 남아 있을 때 사용한다
  - 근거 없는 추정은 모두 이 단계로 보낸다

- `high`
  - 공고 원문 근거가 있는 강한 위험 신호가 핵심 축에서 반복적으로 확인되고
  - 그 위험 신호가 반복업무, 권한 부재, 성과 기준 부재, 성장 구조 부재, 전문성 희석 등 핵심 판정축에 걸쳐 누적될 때 사용한다

## 9. 최종 위험도 결정 원칙

- 최종 위험도는 축 개수만 기계적으로 세지 말고 핵심 판정축의 중요도를 반영한다.
- `needs_verification` 항목이 하나라도 있다고 자동으로 최종 위험도를 `needs_verification`으로 보내지 않는다.
- 대신 핵심 판단에 영향을 주는 `needs_verification`이 남아 있을 때 `needs_verification`을 우선한다.
- `insufficient`가 일부 있어도, 핵심 축의 긍정 근거가 충분히 닫혀 있으면 `low`는 가능하다.
- 반대로 비핵심 축이 좋아 보여도, 핵심 축의 위험 근거가 닫혀 있으면 `high`가 가능하다.
- 근거 없이 `low` 또는 `high`를 주는 것은 금지한다.

## 10. 핵심 축과 보조 체크의 역할

### 10.1 핵심 축

최종 위험도는 아래 7대 핵심 판정축을 중심으로 정한다.

- 반복업무 비중
- 의사결정 권한 및 자율성
- 성과 측정 가능성
- 난이도 상승 가능성
- 시장 수요가 있는 범용 역량
- 업무 범위 명확성 / 직무 전문성 집중도
- 학습 / 피드백 구조

### 10.2 보조 체크

아래 항목은 보조 체크다.

- 고용 형태
- 조직 내 포지션
- 기술 및 프로세스 낙후도

보조 체크 적용 원칙:

- 계약직, 프리랜서, 인턴 등 고용 형태만으로 최종 위험도를 `high`로 올리지 않는다.
- 고용 형태나 조직 내 포지션은 단독 판정 근거가 아니라 보조 근거다.
- 이 항목들은 반복업무, 성과 기준 부재, 권한 불명확성 같은 핵심 위험 신호와 함께 나타날 때만 최종 위험도 판단에 보조적으로 반영한다.
- 공고에 고용 구조가 불명확하면 "고용 구조 확인 필요" 질문으로 연결한다.

## 11. 근거 규칙

- `keyEvidence.quote`는 공고 원문에 실제 있는 문장이어야 한다.
- `interpretation`은 `quote`를 넘어서 과장하면 안 된다.
- `whyImportant`는 커리어 리스크와의 연결 설명만 한다.
- 회사 정보는 직무와 연결될 때만 사용한다.
- 회사 정보는 핵심 축의 직접 근거보다 우선하지 않는다.
- 확인되지 않은 추정은 최종 결론 근거로 사용하지 않는다.

## 12. 출력 계약

### 12.1 무료 결과

- 한 줄 결론
- 가장 강한 근거 1개
- 짧은 이유 2~3개
- 확인 질문 1개

무료 결과 원칙:

- 무료 결과는 짧게 유지한다.
- 상세 7축 전체를 무료 surface에 과다 노출하지 않는다.
- 긍정이든 위험이든 근거 없는 단정은 금지한다.

### 12.2 유료 결과

- 최종 요약 1문장
- 7대 핵심 판정축
- 보조 체크 항목
- 핵심 근거 3~5개
- 면접 질문 5~7개
- 답변 해석 가이드
- 최종 행동 가이드

## 13. 실패 등급

### 13.1 출시 보류급 실패

아래 중 하나라도 발생하면 출시 보류급으로 본다.

- 공고에 없는 내용을 사실처럼 단정
- 근거 없는 `high` 판정
- 근거 없는 `low` 판정
- `keyEvidence.quote`가 원문에 없음
- 위험도 체계가 `low / needs_verification / high` 3단계를 벗어남
- `evidenceStatus`와 `signalDirection`을 분리하지 못해 해석이 왜곡됨
- 고용 형태만으로 최종 위험도를 `high`로 올림
- 핵심 축의 강한 확인 필요 신호를 놓쳐 잘못된 안심 결론을 냄

### 13.2 출시 후 개선 가능 실패

아래는 출시 후 개선 가능으로 본다.

- 표현이 다소 장황함
- 축별 문장 톤이 덜 일관적임
- 질문 우선순위가 조금 아쉬움
- `low`와 `needs_verification` 경계에서 보수적으로 흔들림
- 상세 설명의 설득력 차이
- 같은 의미를 조금 다른 문장으로 표현함

## 14. 수익화 점검 항목

아래 항목은 분석 정확도 실패와 분리해서 수익화 점검 항목으로 본다.

- 무료 결과가 유료 가치 핵심을 과도하게 노출하는가
- 무료 결과가 짧고 날카로운가
- 유료 결과가 단순히 더 긴 설명이 아니라 더 구체적인 확인 기준을 주는가

## 15. 테스트 체크포인트

- 모든 결과는 허용된 `evidenceStatus` 값만 사용한다.
- 모든 결과는 허용된 `signalDirection` 값만 사용한다.
- 최종 위험도는 `low / needs_verification / high`만 사용한다.
- 근거 없는 축 판단은 `needs_verification` 또는 `insufficient` 계열로 남아야 한다.
- `quote`는 원문 substring이어야 한다.
- 고용 형태 단독 신호만으로 `high`가 나오지 않아야 한다.
- 핵심 축의 강한 확인 필요 신호가 남아 있으면 성급한 `low`가 나오지 않아야 한다.
- 고위험 샘플, 혼합 샘플, 낮은 위험 샘플을 분리 검증한다.

## 16. 현재 내부 상태와의 호환 원칙

현재 내부 엔진은 더 세분화된 상태를 유지할 수 있다.

다만 외부 contract 해석은 아래 방식으로 정리한다.

- 내부 세부 상태는 필요하면 유지한다.
- 하지만 API contract, 프롬프트 규칙, 테스트 기대값에서는 `evidenceStatus`와 `signalDirection`을 분리해 해석 가능해야 한다.
- 같은 `confirmed`라도 `positive`일 수도 있고 `risk`일 수도 있으며, 두 경우는 최종 위험도 계산에서 다르게 반영해야 한다.

## 17. 아직 열어둬야 할 결정

이 초안에서 아직 구현 전에 확정해야 할 부분은 아래다.

- 7대 핵심 축 중 최종 위험도 계산에서 더 무겁게 볼 축의 우선순위
- `confirmed + mixed`를 최종 위험도 계산에서 어떻게 가중할지
- 무료 결과에 `evidenceStatus`와 `signalDirection`을 직접 노출할지, 내부용으로만 유지할지
- 기존 `fiveAxes` 호환 구조에 이 계약을 어떻게 얹을지
